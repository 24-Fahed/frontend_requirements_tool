# v0.2.0 交互式模拟器独立 Document 重构草案

## 1. 背景

当前交互式需求收集子系统已支持：

- 标准 Vue SFC 动态加载
- Vant 组件运行
- 基于 `vuedraggable` 的组件面板拖入与容器排序

但现有实现存在一类结构性问题：

1. 手机模拟器、组件面板、运行时组件都运行在主文档中。
2. Vant 的 `fixed`、`teleport`、overlay、dropdown 等能力默认面向浏览器主视口，而不是手机模拟器视口。
3. 组件面板虽然概念上属于交互式子系统，但入口和显隐控制仍由界面子系统承担。
4. 当前“界面子系统”和“交互式需求收集子系统”的运行时边界与文档边界并不一致。

本次重构拟将交互式模拟器升级为一个相对独立的运行时文档，使其内部自行承载：

- 模拟器运行时渲染
- 组件面板
- 组件预览
- 拖拽闭环
- 编辑/预览切换
- 面板开关入口

## 2. 重构目标

### 2.1 功能目标

1. 手机模拟器运行在独立 `document` 中。
2. 组件面板迁入模拟器内部，由模拟器状态栏按钮控制开关。
3. 面板拖拽与模拟器容器拖拽在同一 `document` 中闭环执行。
4. Vant 组件在预览和编辑场景下均以模拟器文档作为运行时视口。
5. 需求收集表格与保存结果链路保持可用。

### 2.2 架构目标

1. 交互式需求收集子系统成为“独立编辑器运行时”。
2. 界面子系统降级为宿主与导航层，不再参与交互式编辑器内部控制。
3. 明确宿主接口与编辑器接口边界，降低跨子系统状态耦合。

### 2.3 非目标

本次重构不包含以下内容：

1. 后端 API 协议变更。
2. 需求配置文件格式变更。
3. 默认布局 JSON 结构变更。
4. 组件属性编辑器的大规模重做。
5. Python 后端实现重写。

## 4. 现状问题总结

### 4.1 运行时边界问题

当前 `PhoneFrame.vue`、`ComponentPanel.vue`、运行时组件和 Vant 样式均在主文档中运行。导致：

- 手机壳只是视觉容器，不是真正的运行时容器。
- `position: fixed` 元素相对浏览器视口而非模拟器视口定位。
- `teleport` 与 overlay 默认落在主文档层级。

### 4.2 子系统边界问题

当前界面子系统仍负责：

- 组件面板入口
- 组件面板显隐状态
- 模拟器和面板的共同装配

这与“组件面板属于交互式子系统”的设计描述不完全一致。

### 4.3 交互模型问题

虽然现有拖拽已迁移到 `vuedraggable`，但它依赖“主文档中的面板”和“主文档中的模拟器容器”共享同一拖拽上下文。一旦需要支持真正独立的运行时文档，就必须重构交互闭环。

## 5. 新架构设计

## 5.1 总体原则

1. 界面子系统只负责宿主布局，不负责交互式编辑器内部行为。
2. 交互式需求收集子系统在独立文档中闭环完成编辑器行为。
3. 主文档与模拟器文档之间只通过明确的宿主桥接层同步状态。
4. 组件加载器必须支持“目标文档上下文”。

### 5.2 新的子系统边界

#### 界面子系统

保留职责：

- 整体页面布局
- 左侧分类导航
- 中央需求表格显示
- 模拟器区域显示/隐藏
- 全局结果保存

移除职责：

- 打开/关闭组件面板
- 渲染组件面板
- 管理交互式编辑器内部拖拽

#### 交互式需求收集子系统

新增/强化职责：

- 创建和维护模拟器独立文档
- 在独立文档中渲染编辑器界面
- 组件面板与组件预览
- 模拟器内部拖拽交互
- 编辑/预览模式切换
- 运行时组件加载和样式注入

### 5.3 新的运行时分层

建议形成如下结构：

1. `PhoneFrameHost`
作用：
- 宿主组件
- 创建 iframe
- 负责宿主文档与模拟器文档的桥接

2. `SimulatorDocumentApp`
作用：
- 挂载到 iframe 的根 Vue 应用
- 组织模拟器状态栏、画布、面板

3. `SimulatorTopBar`
作用：
- 模拟器状态栏
- 提供“预览/编辑切换”“组件面板开关”等入口

4. `SimulatorCanvas`
作用：
- 承载节点树渲染
- 承接拖拽容器

5. `SimulatorPalette`
作用：
- 模拟器内部组件面板
- 组件/布局预览与拖出

6. `RuntimeComponentLoader`
作用：
- 在指定文档上下文中加载 Vant 和 SFC 样式

## 6. 接口调整方案

### 6.1 保持不变的接口

以下接口语义保持不变：

- `/api/requirements`
- `/api/components`
- `/api/default-layout`
- `/api/results`
- `simStore.loadDefault`
- `simStore.reset`
- `simStore.getResult`

### 6.2 需要收敛或删除的接口

1. `appStore.panelOpen`
处理：
- 从应用级共享状态中移除
- 改为交互式子系统内部状态

2. `I_Navigation.togglePanel`
处理：
- 从界面子系统接口中删除
- 面板开关改由模拟器内部按钮控制

3. `SimulatorArea` 直接渲染 `ComponentPanel`
处理：
- 删除
- 宿主只渲染 `PhoneFrameHost`

### 6.3 新增接口建议

建议在交互式子系统内部形成宿主桥接接口：

- `mountSimulatorDocument(targetFrame)`
- `syncSimulatorContext(payload)`
- `syncSimulatorTree(nodes)`
- `disposeSimulatorDocument()`

这些接口初期可先以内聚的 service 形式存在，不必先上复杂消息总线。

## 7. 数据流设计

### 7.1 初始化

1. 主应用加载需求配置、组件清单、默认布局、已保存结果。
2. `SimulatorArea` 挂载 `PhoneFrameHost`。
3. `PhoneFrameHost` 创建 iframe 文档。
4. 交互式子系统在 iframe 中挂载 `SimulatorDocumentApp`。
5. 主应用将 `componentList`、`layoutList`、`simStore` 当前状态同步给 iframe。

### 7.2 打开交互式问题

1. `QuestionRow` 根据问题加载默认布局或历史结果。
2. `simStore` 更新 `nodeTree` 和 `currentQuestionId`。
3. `PhoneFrameHost` 侦听状态变化并同步到模拟器文档。
4. iframe 内编辑器立即渲染最新布局。

### 7.3 拖拽编辑

1. 用户在模拟器内部打开组件面板。
2. 组件面板中的 `vuedraggable` 与画布中的 `vuedraggable` 共享同一 document 和 group。
3. 拖拽结果直接写入 `simStore.nodeTree`。
4. 宿主与模拟器文档不需要在拖拽过程中来回传输 HTML5 DnD 事件。

### 7.4 保存结果

1. 需求收集子系统仍从 `simStore.getResult()` 获取结构化结果。
2. 保存格式保持 `{ nodes: LayoutNode[] }` 不变。

## 8. 对 SOLID 的改进说明

### 8.1 单一职责原则

改造后：

- `AppHeader` 只处理全局界面导航
- `SimulatorArea` 只作为宿主容器
- `PhoneFrameHost` 只负责宿主与独立文档桥接
- `SimulatorDocumentApp` 只负责编辑器运行时

比当前“主界面负责一部分编辑器内部行为”的结构更符合 SRP。

### 8.2 开闭原则

在独立文档中运行后：

- 新增 Vant 组件或其他移动端组件时，不再需要持续修改主文档兼容样式
- 编辑器可以在自身边界内扩展

### 8.3 接口隔离原则

界面子系统不再依赖“打开面板”这种编辑器内部细节接口，只依赖“显示/隐藏模拟器”这类宿主级接口。

### 8.4 依赖倒置原则

高层界面模块不再直接操纵编辑器内部状态，而是依赖宿主级抽象；编辑器实现细节被封装在交互式子系统内部。

## 10. 风险与控制

### 10.1 主要风险

1. iframe 内 Vue 应用挂载与样式注入顺序出错。
2. `vuedraggable` 迁入独立文档后，拖拽容器样式或命中区域异常。
3. 组件加载器缓存如果仍然只按主文档维度缓存，可能导致样式注入丢失。
4. 文档更新不完整，导致设计与实现再度分叉。

### 10.2 控制措施

1. 先完成 `v0.2.0` 草案，再执行代码修改。
2. 将组件加载器改为显式接收运行时上下文。
3. 分步骤删除旧逻辑，避免双系统长期并存。
4. 最终同步更新 `.md` 与 `.puml`，把设计收束回正式文档。
