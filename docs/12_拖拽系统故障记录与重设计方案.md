# 12 拖拽系统故障记录与重设计方案

## 1. 故障记录

### 1.1 文件丢失（根因）

`PhoneFrame.vue` 被删除，`SimulatorArea.vue` 错误引用了不存在的 `PhoneFrameRuntime.vue`。导致模拟器无法渲染，所有拖拽事件无接收者。

### 1.2 未使用已有 Store

重写 PhoneFrame 时自建了独立的 `nodeTree`，没有使用已有的 `simulator.js` store。导致默认布局加载后数据写入 simStore，但 PhoneFrame 读不到。

### 1.3 el-overlay 拦截事件

Element Plus 的 `<el-drawer>` 即使设置 `:modal="false"`，仍会创建全屏 `.el-overlay` 遮罩。从面板拖出时，鼠标经过 overlay 而非 phone-screen，dragover 事件被 overlay 吞掉。

### 1.4 pointer-events: none 反而取消拖拽

尝试对 overlay 设置 `pointer-events: none` 来穿透事件。但拖拽源是 overlay 的子元素，父级变透明后浏览器直接终止了 drag session（dragstart → 立即 dragend）。

### 1.5 dataTransfer.getData() 在 dragover 中返回空

HTML5 DnD 安全限制：`dataTransfer.getData()` 在 `dragover`/`dragenter` 期间可能返回空字符串（Firefox/Safari）。导致先读 payload 再决定是否 `preventDefault()` 的逻辑失效——payload 为 null 时直接 return，`preventDefault()` 永远执行不到。

**修复：** 先无条件 `preventDefault()`，再读 payload。

### 1.6 effectAllowed 与 dropEffect 不匹配

内部拖拽时 `handleDragStart` 设置 `effectAllowed = 'move'`，但 `handleDragOver` 设置 `dropEffect = 'copy'`。HTML5 DnD 规范要求 dropEffect 必须在 effectAllowed 范围内，否则浏览器直接拒绝触发 drop 事件。

**修复：** `dropEffect = effectAllowed === 'move' ? 'move' : 'copy'`

### 1.7 缺少目标位置索引

`simStore.moveNode(nodeId, targetParentId, index)` 的 index 参数从未被传递，默认插入到父节点末尾。导致：
- 拖拽 A 到 B 的位置，期望 [B, A, C]，实际得到 [B, C, A]
- 无法精确重排序

## 2. 故障根因总结

| 编号 | 根因 | 教训 |
|------|------|------|
| 1-2 | 不了解已有数据流就动手写 | 先读懂 store 再写组件 |
| 3-4 | 假设 overlay 是问题就急着修 | 应先验证模拟器内部拖拽是否正常 |
| 5-6 | 不了解 HTML5 DnD 规范细节 | getData 限制、effect 匹配规则 |
| 7 | 只实现了"添加到容器"，没实现"插入到位置" | 需要从设计层面解决索引问题 |

## 3. 重设计方案

### 3.1 核心思路

将拖拽分为两层独立处理：

- **外部拖入**（面板 → 模拟器）：document 级别处理器，绕过 overlay
- **内部操作**（模拟器内移动/重排）：SimNode 级别处理器，直接操作 DOM

两层共享同一套 `applyDragPayload → simStore` 数据流，区别只在事件入口。

### 3.2 插入位置计算

不再依赖"拖到哪个节点上"来判断目标，而是用**鼠标坐标与子元素位置的比较**计算插入索引。

```
VStack 容器内有 3 个子组件：[A, B, C]

鼠标拖拽到 A 和 B 之间：
  A 的 bottom < mouseY < B 的 top
  → 插入索引 = 1（B 的位置，B 及之后后移）

鼠标拖拽到 C 下方：
  mouseY > C 的 bottom
  → 插入索引 = 3（末尾追加）
```

**算法：**
```
for each child element of container (从上到下):
  childCenterY = child.top + child.height / 2
  if mouseY < childCenterY:
    return childIndex   // 插在这个子节点前面
return children.length   // 鼠标在所有子节点下方，追加到末尾
```

### 3.3 视觉指示器

在 `dragover` 期间，在计算出的插入位置显示一条蓝色水平线（drop indicator），让用户明确知道组件将放在哪里。

- 指示器为一个绝对定位的 `<div>`，高度 2px，宽度 100%
- 在容器节点内渲染，位于两个子节点之间或首尾
- `dragleave` 时移除指示器

### 3.4 索引修正（同容器内移动）

当被拖拽节点和目标位置在**同一父容器**内时，先从树中拔出节点会导致后续索引偏移：

```
原始：[A, B, C]，拖 A 到 C 后面
1. 拔出 A → [B, C]，此时 C 的索引从 2 变成 1
2. 目标索引原始是 3（末尾），拔出后应为 2
3. 规则：if sourceIndex < targetIndex, targetIndex -= 1
```

由 `simStore.moveNode` 内部处理此修正。

### 3.5 事件流设计

#### 外部拖入（面板 → 模拟器）

```
DrawerPanel.dragstart
  → dataTransfer.setData({kind:'palette-item', sourceType, sourceName})
  → document.dragover [注册在 document 上]
    → isOverPhoneScreen? → preventDefault()
    → findDeepestContainer(x, y) → 计算 insertIndex
    → 显示 drop indicator
  → document.drop
    → simStore.addNode(containerId, source, insertIndex)
  → document.dragend
    → 清理
```

#### 内部移动（模拟器内重排）

```
SimNode.dragstart
  → dataTransfer.setData({kind:'existing-node', nodeId})
  → SimNode(容器).dragover
    → preventDefault()
    → 计算 insertIndex（基于鼠标 Y 与子元素位置）
    → 显示 drop indicator
  → SimNode(容器).drop
    → simStore.moveNode(nodeId, containerId, insertIndex)
```

#### 非容器节点的作用

非容器节点不作为 drop 目标。当鼠标经过非容器节点时：
- dragover：preventDefault 后冒泡到父容器（由父容器计算位置）
- drop：冒泡到父容器处理

### 3.6 组件解耦分析

当前实现与具体组件**无强依赖**。判断依据：

| 依赖点 | 是否组件相关 |
|--------|-------------|
| `isContainer` 判断 | 通用，基于 `Array.isArray(node.children)` |
| 组件解析 | 通用，通过 meta.name 从 registry 查找 |
| props 规范化 | `normalizedProps` 处理了 items/number/boolean，是通用规则 |
| 拖拽数据 | 只传递 nodeId 或 sourceName，不涉及组件内部结构 |
| 插入位置计算 | 基于子元素的 DOM 位置（getBoundingClientRect），与组件类型无关 |

**结论：** 重设计只需修改事件处理和索引传递逻辑，不需要修改组件加载/渲染部分。

- simStore 的 addNode / removeNode / findNode / extractFromTree
- SimNode 的渲染逻辑（resolveRuntimeComponent + normalizedProps）
- DrawerPanel 的 dragstart 和面板 UI
- document 级别 dragover/drop 的注册和坐标判断
