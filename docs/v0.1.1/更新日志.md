# v0.1.1 更新日志 - 拖拽系统重构

## 1. 版本概述

**版本号**：v0.1.1
**基线版本**：v0.0.0
**变更范围**：前端拖拽交互系统（DrawerPanel、PhoneFrame、SimNode）
**变更性质**：内部重构，不改变外部接口和数据格式

## 2. 问题分析

v0.0.0 的拖拽系统使用原生 HTML5 Drag and Drop API 实现，经历了 7 次故障（详见 `docs/12_拖拽系统故障记录与重设计方案.md`）。归纳为以下根本问题：

### 2.1 抽象层选择错误

直接使用 HTML5 DnD API 意味着要自行处理所有浏览器底层细节：
- `dataTransfer.getData()` 在 `dragover` 期间可能返回空（Firefox/Safari 安全限制）
- `effectAllowed` 与 `dropEffect` 必须严格匹配，否则浏览器拒绝触发 `drop`
- 各浏览器行为不一致

### 2.2 Element Plus el-drawer overlay 拦截事件

`el-drawer` 即使设置 `:modal="false"` 仍会创建全屏 `.el-overlay` 遮罩层。从面板拖出时，鼠标经过 overlay 而非 phone-screen，dragover 事件被 overlay 吞掉。当前方案用 document 级监听器绕过，但增加了复杂性。

### 2.3 索引计算和同容器移动

当前方案需要自行实现鼠标坐标与子元素位置的比较算法来计算插入索引，以及在同容器移动时进行 `sourceIndex < targetIndex` 的修正。这些逻辑容易出错且难以维护。

## 3. 设计决策

### 决策一：使用 vuedraggable（Sortable.js）替代原生 HTML5 DnD

**理由**：
- vuedraggable 已在 `package.json` 中声明（`"vuedraggable": "^4.1.0"`），但代码中完全未使用
- Sortable.js 内部处理了所有 HTML5 DnD 的浏览器兼容性问题
- 自动提供精确的插入位置索引，无需手动计算
- 同容器移动由 Sortable.js 原子操作处理，不存在索引偏移问题
- 内置视觉反馈（ghost 元素、插入占位符、拖拽动画）
- 原生支持触屏设备

**映射关系**：

| 当前实现 | vuedraggable 方案 |
|---------|------------------|
| `dataTransfer.setData/getData` | `:list` 数据绑定 + `:clone` 回调 |
| `effectAllowed / dropEffect` 手动匹配 | Sortable.js 内部处理 |
| document 级 dragover/drop 绕过 overlay | 不使用 el-drawer，问题不存在 |
| 自行计算插入索引 | Sortable.js 自动提供 |
| `sourceIndex < targetIndex` 索引修正 | 原子操作，无需修正 |
| swapNodes（落在非容器上交换） | 自然排序：所有容器（含根）接受插入 |
| 手动 drop indicator | Sortable.js 内置 ghost/placeholder |

### 决策二：用自定义滑出面板替代 el-drawer

**理由**：
- overlay 拦截事件是 el-drawer 的固有问题，无法从外部彻底解决
- 自定义面板仅需约 30 行代码（CSS transition + fixed 定位）
- 没有 overlay = 没有事件穿透问题，拖拽路径上无任何遮挡物
- 视觉效果与 el-drawer 相同（从右侧滑出）

### 决策三：调整交互语义

| 旧行为 | 新行为 | 理由 |
|--------|--------|------|
| 落在非容器节点上 → swap 交换位置 | 非容器节点不接受 drop，由父容器处理插入位置 | 更符合主流拖拽排序的直觉 |
| 落在容器节点上 → 添加到容器末尾 | 落在容器中 → 插入到鼠标所在位置 | 精确控制插入点 |
| 面板拖出 → addNode(source) | 面板拖出 → clone 回调创建新 LayoutNode | 由 vuedraggable 的 clone 机制处理 |

## 5. 不变的部分

以下部分**不做任何修改**：

- `simulator.js` (Pinia Store)：核心 API（addNode, removeNode, moveNode, loadDefault, reset, getResult）保持不变
- `componentLoader.js`：动态组件加载机制不变
- `app.js` (Store)：应用状态管理不变
- `TableRenderer.vue` / `QuestionRow.vue`：需求收集流程不变
- `AppHeader.vue` / `SideBar.vue` / `ContentArea.vue` / `SimulatorArea.vue`：界面布局不变
- 后端代码（Python/FastAPI）：完全不受影响
- 数据格式（requirements.yaml, components.json, default_layout.json, result.json/yaml）：完全兼容
- 子系统接口（I_Simulator, I_Navigation, I_ParseService 等）：接口签名不变

## 6. vuedraggable 配置方案

### 6.1 面板（palette）配置

```
group: { name: 'simulator', pull: 'clone', put: false }
clone: paletteItem → LayoutNode 转换
```

`pull: 'clone'` 确保从面板拖出时原件保留。`clone` 回调负责将 ComponentInfo 转换为 LayoutNode（生成唯一 ID、设置默认 props、标记容器子数组）。

### 6.2 模拟器根列表配置

```
group: { name: 'simulator', pull: true, put: true }
:list: simStore.nodeTree（直接绑定 Pinia 响应式数组）
```

### 6.3 容器节点子列表配置

```
group: { name: 'simulator', pull: true, put: true }
:list: node.children（直接绑定响应式子节点数组）
```

三个 `<draggable>` 共享同一个 group name `'simulator'`，实现跨列表自由拖拽。

## 7. 数据流对比

### 旧数据流

```
DrawerPanel.dragstart
  → dataTransfer.setData(JSON payload)
  → document.dragover (绕过 overlay)
  → document.drop
    → decodeDragPayload()
    → findDropTarget(x, y)
    → applyDragPayload()
      → simStore.addNode() / simStore.moveNode() / simStore.swapNodes()
```

### 新数据流

```
DrawerPanel <draggable pull="clone">
  → Sortable.js 内部处理拖拽
  → clone 回调生成 LayoutNode
  → Sortable.js 自动插入到目标 :list 数组
  → Vue 响应式自动更新 UI
```

节点在容器间移动时，Sortable.js 原子操作完成"从源数组移除 + 插入目标数组"，无需手动调用 store 方法。
